## 什麼是 OpenSpec

OpenSpec 是一套「規範驅動開發」(Spec-Driven Development) 的 AI 協作框架。核心概念：**在 AI 動手改程式碼之前，先產出可審閱的規格書（spec）**，讓你確認範圍正確後再執行。

適合場景：
- 模組架構複雜、跨多資料夾（如 XOOPS）
- 老舊框架有大量慣例，AI 不熟悉容易出錯
- 需求持續追加、沒有文件就失控

## 安裝步驟

### 前置條件

- Node.js 18+（`node -v` 確認）
- 專案已有 git repo

### Step 1：在 XOOPS 模組根目錄初始化

```bash
cd /path/to/your/xoops-module
npx openspec init
```

產生的目錄結構：

```
your-module/
├── specs/
│   ├── openspec.config.md    ← AI 工作指引（自動生成，勿手動刪）
│   └── convention.md         ← 主規格書（你要填寫的）
├── changes/                  ← 每次功能的 delta spec 存放處
└── ... (原有程式碼)
```

### Step 2：填寫主規格書

複製 [[OpenSpec-XOOPS-convention]] 範本的內容到 `specs/convention.md`，依你的模組實際情況調整。

**這是最重要的一步**，後續所有 AI 協作都依賴這份文件。

### Step 3：驗證設定

```bash
npx openspec check
```

確認 OpenSpec 可以正確讀取 specs/ 目錄。

## 每次開發新功能的流程

```
/explore
  └─ AI 向你確認「這次要改什麼、影響哪些檔案」
     確認清楚後繼續

/propose
  └─ 產出 changes/feature-名稱/proposal.md
     ├─ 目標說明
     ├─ 影響範圍（ADD / MODIFY / DELETE）
     └─ 不影響範圍（明確排除）

     ← 你審閱 proposal.md，確認範圍正確再繼續

/apply
  └─ 依 proposal.md 執行，每完成一個 task 就 git commit

/archive
  └─ 將本次 delta spec 歸檔
     └─ 更新主 specs/convention.md（如有新發現的慣例）
```

### 黃金原則

| 操作 | 要做的事 |
|------|---------|
| `/propose` 後 | **先讀 proposal.md**，確認範圍對了再 `/apply` |
| `/apply` 中 | 每完成一個 task 就 `git commit`，方便回滾 |
| 發現新慣例 | 立刻加進 `specs/convention.md`，不要等 `/archive` |
| 有疑問 | 先 `/explore` 再次確認，不要讓 AI 猜 |

## 常見問題

**Q: 可以在現有模組（已有程式碼）上使用嗎？**

可以。`npx openspec init` 不會修改現有程式碼，只新增 `specs/` 和 `changes/` 目錄。

**Q: specs/convention.md 要多詳細？**

夠讓 AI「不猜測」為止。重點是寫「禁止事項」和「命名規則」，AI 犯過的錯都值得記錄。

**Q: 每個功能都要跑完整流程嗎？**

修 bug 可縮短為 `/explore` → `/apply`（跳過 `/propose`）。新功能建議跑完整流程。
