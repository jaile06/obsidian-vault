## 定義

任務拆解（Task Decomposition）是把寫給人看的 Human SOP，轉換成 agent 真正能穩定執行的 Agentic Workflow 的方法論。核心思路是把一個大而模糊的任務拆成多個輸入輸出明確、可獨立除錯的小節點，而非丟給單一「mega agent」整包處理。

## 核心重點

- **三層名詞**：
  - **Human SOP**：寫給人看的流程文件，依賴讀者腦中的默會知識（Tacit Knowledge）補完細節
  - **Skill**：把方法論、判斷標準打包成 agent 可重用的執行單位（含 SKILL.md、references、scripts），對應單一任務
  - **Agentic Workflow**：由多個 agent / tools / skills 組成的完整生產線
- **mega agent 為何失敗**：把整包模糊任務丟給單一最強模型，產出不可預測、無法 review、出錯難定位；企業需要的是穩定性、可觀測性、可修復性
- **四步方法論**：
  1. **格式標準化**：參數化（如 mode/temperature 取代寫死的設定）、用 RFC 2119 的 MUST/SHOULD/MAY 定義規則強度、用 Markdown 結構化 Parameters/Steps/Error Handling
  2. **任務拆解與連結**：拆成 pipeline 中各自獨立的 step，各有明確 input/output，靠「artifact」（如 JSON）串接上下游 agent
  3. **雙向開發**：第一版 SOP 必有疏漏，因為默會知識要等 agent 出錯才會被發現；透過小步快跑、實際執行→發現問題→補規則→再跑的迭代循環收斂
  4. **整合與執行環境**：接上真實工具（資料庫、API、[[MCP]]），並設計 human-in-the-loop checkpoint，在高風險決策點讓 agent 停下來等人確認

## 不同視角 / 矛盾之處

- 模型能力越來越強，但「任務本身太大太模糊」才是表現不佳的主因，而非模型或 prompt 不夠好
- 完美主義式的一次到位 SOP（如花兩個月寫的「完美 SOP」一跑就垮）不如小步快跑、多輪迭代
